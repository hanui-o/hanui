Claude Code 시리즈 6편이에요. [5편에서 MCP로 노션·피그마를 연동](/blog/claude-code-mcp)했는데요.

오늘은 **백엔드 개발에 Claude Code를 쓰는 법**이에요.

"Claude Code는 프론트엔드용 아닌가?" 싶을 수 있는데, 아니에요. 저는 지금 hanui CMS 백엔드를 Claude Code로 짜고 있어요. Spring Boot + 전자정부프레임워크 v5예요.

## 백엔드에서 Claude Code가 특히 유용한 이유

프론트엔드는 결과가 눈에 보여요. 근데 백엔드는 다 타이핑이에요.

- DB 스키마 → SQL 파일 → 마이그레이션
- API 명세 → Controller → Service → Repository → Mapper
- 반복되는 CRUD 패턴

이 패턴 반복 작업에서 Claude Code가 빛을 발해요.

## 실제 워크플로우

제가 hanui CMS 백엔드를 만들면서 쓴 방식이에요.

### 1. 노션 기획서 → DB 스키마

```
나: 노션 기획서 보고 DB 스키마 만들어줘
    https://notion.so/...

Claude: (노션 직접 읽음)
        → ERD 분석
        → PostgreSQL DDL 생성
        → Flyway 마이그레이션 파일 생성
```

노션 MCP가 연결돼 있으면 기획서를 직접 읽어요. 복붙 없이 바로 스키마가 나와요.

결과:

```sql
-- V1__init_schema.sql
CREATE TABLE users (
    id          BIGSERIAL PRIMARY KEY,
    login_id    VARCHAR(50) UNIQUE NOT NULL,
    password    VARCHAR(255) NOT NULL,
    name        VARCHAR(50) NOT NULL,
    email       VARCHAR(100),
    is_active   BOOLEAN DEFAULT true,
    created_at  TIMESTAMP DEFAULT now(),
    updated_at  TIMESTAMP DEFAULT now()
);

CREATE TABLE posts (
    id          BIGSERIAL PRIMARY KEY,
    board_id    BIGINT NOT NULL REFERENCES boards(id),
    title       VARCHAR(300) NOT NULL,
    content     TEXT,
    status      VARCHAR(20) DEFAULT 'DRAFT',
    is_pinned   BOOLEAN DEFAULT false,
    view_count  INTEGER DEFAULT 0,
    created_by  BIGINT NOT NULL REFERENCES users(id),
    created_at  TIMESTAMP DEFAULT now(),
    updated_at  TIMESTAMP DEFAULT now(),
    deleted_at  TIMESTAMP
);
-- ...
```

### 2. API 명세 → Controller 한 번에

CRUD API 명세를 주면 Controller부터 Repository까지 한 번에 만들어줘요.

```
나: 게시글 API 명세 보고 구현해줘
    - GET /api/v1/boards/:boardCode/posts (목록, 페이지네이션)
    - POST /api/v1/admin/posts (등록, EDITOR 이상)
    - PUT /api/v1/admin/posts/:id (수정)
    - DELETE /api/v1/admin/posts/:id (소프트 삭제)

Claude: PostController.java
        PostService.java
        PostRepository.java (MyBatis)
        PostMapper.xml
        PostDto.java (Request/Response)
        → 전부 생성
```

하나씩 만들면 30분 걸릴 걸 5분에 끝내요.

### 3. CLAUDE.md로 일관성 유지

백엔드에서 CLAUDE.md가 더 중요해요. 코드 패턴이 미묘하게 달라지면 나중에 유지보수가 힘들거든요.

저는 이렇게 써뒀어요:

```markdown
## 백엔드 코드 패턴

### 응답 형식
모든 API는 ApiResponse<T>로 감싸서 반환:
{ "success": true, "data": {...} }
{ "success": false, "error": { "code": "...", "message": "..." } }

### 계층 구조
Controller → Service → Repository (MyBatis Mapper)
- Controller: 요청/응답 변환만
- Service: 비즈니스 로직, 트랜잭션
- Repository: 쿼리만

### 소프트 삭제
posts, media, pages는 deleted_at 컬럼으로 소프트 삭제
실제 DELETE 쿼리 사용 금지

### 에러 코드
CustomException + GlobalExceptionHandler 패턴 사용
ErrorCode enum에 정의된 코드만 사용
```

이걸 써두면 50개 API가 전부 같은 패턴으로 나와요.

## 반복 작업이 제일 빠르다

백엔드 개발에서 가장 지루한 부분이 CRUD 반복이에요.

```
게시판 API → 게시글 API → 고정페이지 API → 미디어 API → ...
```

구조가 거의 똑같아요. Controller, Service, Repository, Mapper, DTO. 이름만 바뀌고.

Claude Code한테 한 도메인을 완성하고 나서 "이 패턴으로 미디어 API 만들어줘"라고 하면 바로 나와요. 이전에 만든 코드를 참고해서 일관된 패턴으로.

## 테스트 코드도 같이

```
나: PostService 테스트 코드 짜줘

Claude: PostServiceTest.java 생성
        - 게시글 등록 성공/실패
        - 소프트 삭제 확인
        - 권한 검증
        → 케이스별로 다 짜줌
```

이걸 직접 짜면 또 30분인데, 같이 나와요.

## 막히는 부분은 같이 디버깅

Spring Security 설정이나 JWT 필터 같이 복잡한 부분은 Claude Code랑 같이 디버깅해요.

```
나: JWT 필터에서 401이 계속 나와
    (파일 첨부)

Claude: SecurityConfig에서 /api/v1/auth/** 경로를
        permitAll()에 포함했는지 확인해줘
        → (코드 보고) 여기 빠졌네요, 추가할게요
```

에러 로그 붙여넣으면 원인 찾는 것도 빨라요.

## 프론트랑 동시에 개발

백엔드 API가 아직 없어도 프론트 개발을 멈출 필요가 없어요.

1. API 명세 확정
2. 프론트: Mock 데이터로 UI 개발
3. 백엔드: Claude Code로 API 구현
4. 프론트: Mock → 실제 API 연동

저는 hanui CMS 프론트 UI를 먼저 다 만들고, 지금 백엔드를 짜고 있어요. Claude Code 덕에 둘이 거의 동시에 진행할 수 있었어요.

## 정리

백엔드에서 Claude Code가 특히 유용한 건:

| 작업 | 혼자 | Claude Code |
|------|------|-------------|
| DB 스키마 설계 | 2~3시간 | 30분 |
| CRUD API 1개 도메인 | 2시간 | 30분 |
| 테스트 코드 | 1시간 | 같이 나옴 |
| 에러 디버깅 | 막막 | 같이 봄 |

코드를 대신 짜준다기보다, **내가 설계하고 Claude가 구현하는** 느낌이에요. 설계 판단은 제가 하고, 타이핑은 Claude가 해요.

[claude-settings](https://github.com/hanui-o/claude-settings)에 백엔드용 커맨드도 있어요 (`/api`, `/db`, `/migrate`):

```bash
bash <(curl -sL https://raw.githubusercontent.com/hanui-o/claude-settings/main/setup.sh)
```
