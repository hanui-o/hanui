접근성 시리즈 6편이에요. [5편에서 디자인 시스템 컴포넌트 단위 자동화](/blog/a11y-design-system-automation)를 다뤘는데요.

오늘은 한 단계 위 — **운영 중인 서비스를 페이지 단위로 스캔해서 CI에 붙이는 방법**을 공유할게요. 실제로 지금 진행 중인 프로젝트의 구조를 정리한 거예요.

## 왜 페이지 단위 스캔이 필요한가

컴포넌트 단위 a11y 테스트는 "Button에 aria-label이 있나?" 수준이에요. 근데 실제 사용자가 만나는 건 **조합된 페이지**고, 여기서 생기는 문제가 더 많아요:

- heading 레벨이 h1 → h3으로 건너뛰는 경우
- form label과 input 연결이 깨지는 경우
- 모달이 열렸을 때 포커스 트랩이 안 걸리는 경우
- 동적으로 추가된 영역이 `aria-live`로 안 알려지는 경우

**컴포넌트는 통과하는데 페이지는 위반**이 자주 생겨요. 그래서 E2E 레벨에서 한 번 더 훑어야 해요.

## 도구 조합: axe-core + Playwright

```
axe-core (WCAG 규칙 엔진) + Playwright (브라우저 자동화) + GitHub Actions (CI)
```

왜 이 조합인가:

| 도구 | 역할 | 대안 대비 |
|------|------|----------|
| **axe-core** | WCAG 2.1 AA 규칙 검사 | 업계 표준. Deque 개발. false positive 낮음 |
| **Playwright** | 실제 브라우저에서 페이지 로드 | Cypress 대비 멀티 브라우저, 인증 storage state 처리 편함 |
| **GitHub Actions** | PR마다 자동 실행 | 별도 인프라 불필요, artifact 보관 쉬움 |

## 전체 아키텍처

```
PR 생성
  ↓
GitHub Actions 트리거
  ↓
1. Playwright setup — 로그인 storage state 복원
  ↓
2. axe-scan — 대상 페이지마다 axe 실행
  ↓
3. 결과 수집 — 페이지별 violation 수 집계
  ↓
4. PR 코멘트 자동 게시 — 이전 커밋 대비 증감 표시
  ↓
5. Artifact 업로드 — HTML 리포트로 상세 확인 가능
```

## 핵심 1: 인증된 페이지 어떻게 스캔하는가

SSO, OAuth, 이메일 인증 등이 있는 서비스는 테스트 환경에서 자동 로그인이 거의 불가능해요. 답은 **로그인된 상태를 파일로 저장해서 재사용**하는 거예요.

### 로그인 storage state 생성 (로컬)

```bash
pnpm exec playwright codegen \
  --save-storage=playwright/.auth/user.json \
  https://your-qa-domain.com
```

브라우저가 열리면:
1. 수동으로 로그인
2. 브라우저 닫기
3. `user.json`에 쿠키/토큰이 저장돼요

### CI에서 쓰기

이 `user.json`을 GitHub Secret에 통째로 등록하고, CI에서 파일로 복원해서 Playwright가 로드하게 해요.

```ts
// playwright/a11y/auth.setup.ts
import { test as setup } from '@playwright/test';
import fs from 'node:fs';

setup('restore auth', async () => {
  if (process.env.PLAYWRIGHT_AUTH_STATE) {
    fs.writeFileSync(
      'playwright/.auth/user.json',
      process.env.PLAYWRIGHT_AUTH_STATE
    );
  }
});
```

```ts
// playwright.config.ts
export default defineConfig({
  projects: [
    { name: 'setup', testMatch: /auth\.setup\.ts/ },
    {
      name: 'a11y',
      dependencies: ['setup'],
      use: { storageState: 'playwright/.auth/user.json' },
      testMatch: /axe-scan\.spec\.ts/,
    },
  ],
});
```

**주의:** 토큰이 만료되면 `user.json`을 다시 생성해야 해요. 이게 운영상 가장 큰 함정이에요 (뒤에서 다룸).

## 핵심 2: axe 스캔 로직

```ts
// playwright/a11y/axe-scan.spec.ts
import { test } from '@playwright/test';
import AxeBuilder from '@axe-core/playwright';

const PAGES = [
  { name: 'home', url: '/' },
  { name: 'settings', url: '/settings' },
  { name: 'detail', url: '/items/123' },
  // ... 대상 페이지 나열
];

for (const page of PAGES) {
  test(`a11y: ${page.name}`, async ({ page: pw }, testInfo) => {
    try {
      await pw.goto(page.url, { waitUntil: 'domcontentloaded' });
      await pw.waitForTimeout(3000); // 동적 콘텐츠 대기

      const results = await new AxeBuilder({ page: pw })
        .withTags(['wcag2a', 'wcag2aa', 'wcag21a', 'wcag21aa'])
        .analyze();

      console.log(
        `[a11y] ${page.name}: ${results.violations.length} violations`
      );

      // Phase 1 (베이스라인): 수치만 기록, 차단 안 함
      // Phase 2 (게이팅): 아래 주석 해제
      // expect(results.violations).toEqual([]);
    } catch (error) {
      testInfo.annotations.push({
        type: 'scan-failed',
        description: `${page.name}: ${error.message}`,
      });
      console.log(`[a11y] ${page.name}: SCAN_FAILED — ${error.message}`);
    }
  });
}
```

**포인트:**
- `withTags`로 WCAG 2.1 AA까지 검사 (AAA는 너무 빡셈)
- try/catch로 감싸서 **한 페이지 실패가 전체 파이프라인을 막지 않게** 함
- 실패 페이지는 `annotations`로 기록 → 나중에 찾기 쉬움

## 핵심 3: 베이스라인 → 게이팅 2단계 전략

접근성 자동화를 처음부터 "위반 0 강제"로 가면 **아무도 PR을 못 머지해요**. 기존 코드에 위반이 있기 때문이죠.

그래서 2단계로 가요:

### Phase 1 — 베이스라인 (현재 수치 수집)

```ts
// expect(results.violations).toEqual([]); ← 아직 비활성
console.log(`${page.name}: ${results.violations.length} violations`);
```

- 모든 페이지의 violation 수를 **기록만** 함
- PR 코멘트로 이전 대비 증감 표시
- 팀이 "아, 현재 이 페이지는 3개 위반이 있구나" 인식하게 함
- **이 단계에서 2~4주 정도 운영**

### Phase 2 — 게이팅 (위반 증가 시 차단)

```ts
expect(results.violations).toEqual([]);
```

또는 페이지별 기준선 JSON을 만들어서:

```ts
const baseline = { home: 2, settings: 5, detail: 1 };
expect(results.violations.length).toBeLessThanOrEqual(baseline[page.name]);
```

**"더 늘지만 마"** 정책이 가장 현실적이에요. 기존 위반은 별도 개선 작업으로 줄여나가고, CI는 회귀만 막아요.

## 핵심 4: PR 코멘트로 가시성 확보

CI가 초록 체크여도 **실제 violation 수치는 PR 코멘트로 봐야** 해요. 안 그러면 팀이 "통과했네" 하고 넘어가요.

```yaml
# .github/workflows/a11y.yaml
- name: Parse results and post comment
  uses: actions/github-script@v7
  with:
    script: |
      const results = JSON.parse(fs.readFileSync('a11y-summary.json'));
      const body = `## ♿ Accessibility Scan Result

<details>
<summary>페이지별 위반 수</summary>

| 페이지 | 이전 | 현재 | 증감 |
|--------|------|------|------|
${results.map(r => `| ${r.name} | ${r.before} | ${r.after} | ${r.diff} |`).join('\n')}
</details>`;

      github.rest.issues.createComment({
        owner: context.repo.owner,
        repo: context.repo.repo,
        issue_number: context.payload.pull_request.number,
        body,
      });
```

## 실전에서 만나는 함정

### 1. 토큰 만료로 수치가 확 줄어듦

어느 날 갑자기 위반 수가 "페이지당 평균 3~5 → 1"로 떨어지면 **개선된 게 아니라 로그인 실패**예요. 로그인 페이지로 리다이렉트되어 거기서 스캔된 거죠.

**대응:**
- 수치 급감 시 Secret 갱신 체크리스트를 README에 박아두기
- 로컬에서 `--headed`로 돌려서 로그인 상태 눈으로 확인

### 2. 스캔 실패 ≠ 통과 ≠ 위반 없음

3가지 상태를 구분해야 해요:

| 상태 | 의미 | 위반 수치 |
|------|------|----------|
| **통과** | 페이지 로드 성공 + 위반 없음 | 0 (신뢰) |
| **위반 있음** | 페이지 로드 성공 + 위반 발견 | N (신뢰) |
| **스캔 실패** | 페이지 로드 실패 (타임아웃 등) | 0 (❌ 신뢰 불가) |

스캔 실패인데 수치만 보면 "0이네, 통과!"로 착각해요. **annotation으로 명시적으로 기록**하고 Actions 로그에 `SCAN_FAILED`를 남겨서 구별해요.

### 3. CI vs 로컬 대기 전략이 다름

- **CI**: `domcontentloaded + 3초` (빠른 피드백)
- **로컬 baseline**: `networkidle` (더 많이 잡음)

CI가 느슨한 대신 로컬 baseline이 절대 기준선 역할이에요. CI 수치와 로컬 수치가 다르다고 당황하지 마세요.

### 4. 환경 별칭 함정

`dev` 환경이 실제로는 QA 도메인을 가리키도록 aliased 되어 있는 경우가 있어요. 이럴 땐:

- 주석이나 README에 **"왜 이렇게 aliased 됐는지" 명시**
- 실제 환경이 분리되면 `.auth/qa.json`, `.auth/dev.json`으로 파일 나눌 것
- 이 TODO를 코드에 남겨두기

## 실행 방법 요약

### 로컬에서 QA 환경 대상

```bash
BASE_URL=https://qa.your-domain.com \
  pnpm exec playwright test --project=setup --project=a11y
```

### HTML 리포트 보기

```bash
pnpm exec playwright show-report
```

### CI artifact로 상세 확인

```bash
# 최신 run 찾기
gh run list --workflow=a11y.yaml --limit 1

# artifact 다운로드
gh run download {run-id} --name playwright-report --dir /tmp/a11y

# 브라우저로 열기
pnpm exec playwright show-report /tmp/a11y
```

## 마무리

정리하면:

1. **컴포넌트 단위로는 부족해요** — 조합된 페이지에서 생기는 위반이 더 많음
2. **axe-core + Playwright + GitHub Actions** 조합이 가장 가성비 좋음
3. **베이스라인 → 게이팅 2단계 전략**으로 팀이 받아들일 수 있게 함
4. **PR 코멘트로 가시성 확보** — 초록 체크 = 통과 아님을 드러내기
5. **토큰 만료, 스캔 실패 같은 함정**을 README에 박아두기

"자동화 파이프라인 만들었으니 접근성 끝"이 아니에요. 자동화는 **회귀 감지**일 뿐이고, 실제 접근성 개선은 수동 테스트(키보드, 스크린 리더)와 병행해야 해요.

다음 편에서는 **수동 테스트를 체크리스트화**해서 자동화가 못 잡는 영역을 체계적으로 커버하는 방법을 다룰게요.
