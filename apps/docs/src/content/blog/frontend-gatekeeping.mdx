최근 [OpenClaw의 게이트키핑 아키텍처](https://roboco.io/posts/openclaw-gatekeeping-architecture/) 글을 읽었어요. 31만 스타 프로젝트에서 AI 에이전트 8개가 동시에 코드를 쏟아내는 환경의 품질 관리법이에요.

읽으면서 생각했어요. **"이거 규모만 다르지, 프론트엔드 프로젝트에도 똑같이 적용되잖아."**

핵심 원칙은 하나예요.

> **싼 검사는 앞에, 비싼 검사는 뒤에.**

커밋 전에 잡을 수 있는 건 커밋 전에 잡고, CI에서 잡을 건 CI에서 잡고, 사람이 봐야 할 건 마지막에 보는 거예요. 뒤로 갈수록 비용이 비싸지니까요.

## 1단계: 커밋 전 — lint-staged

코드가 저장소에 들어가기 전에 거르는 관문이에요.

```json
// package.json
{
  "lint-staged": {
    "*.{ts,tsx}": ["prettier --write", "eslint --fix"],
    "*.{json,md,css}": ["prettier --write"]
  }
}
```

이 단계에서 잡는 것들:
- 포맷팅 불일치 (탭 vs 스페이스, 세미콜론)
- 사용하지 않는 import
- 기본적인 린트 에러

**비용: 0.** 자동으로 고쳐지니까 개발자가 신경 쓸 게 없어요.

HANUI에서는 `lint-staged` + `husky`로 모든 커밋에 자동 적용하고 있어요. 커밋할 때마다 prettier가 돌아가니까 코드 스타일 논쟁이 사라져요.

## 2단계: 로컬 체크 — 타입 + 빌드

커밋은 됐지만 push 전에 한 번 더 확인하는 관문이에요.

```bash
# 한 줄로 전부 체크
pnpm build
```

프론트엔드에서 `pnpm build`는 생각보다 많은 걸 잡아줘요.

- **TypeScript 타입 에러** — `tsc`가 빌드 과정에서 돌아감
- **import 경로 오류** — 없는 파일 참조하면 빌드 실패
- **환경 변수 누락** — 빌드 타임에 필요한 값이 없으면 실패
- **SSR 호환성** — `window is not defined` 같은 서버 사이드 에러

1단계에서 포맷은 잡았고, 2단계에서 "이 코드가 실제로 돌아가는가"를 확인하는 거예요.

## 3단계: CI — 자동화된 품질 검사

push하면 GitHub Actions가 돌아가요. 여기서부터는 로컬에서 놓친 것들을 잡아요.

```yaml
# .github/workflows/ci.yml
jobs:
  build:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: pnpm/action-setup@v4
      - run: pnpm install --frozen-lockfile
      - run: pnpm build
      - run: pnpm lint
      - run: pnpm test
```

프론트엔드에서 CI가 특히 중요한 이유가 있어요.

**접근성 검사를 자동화할 수 있어요.**

```yaml
- name: 접근성 검사
  run: pnpm exec axe-core --check ./dist
```

WCAG AA 기준 위반이 있으면 CI가 실패해요. 색상 대비 부족, aria 속성 누락, 키보드 접근 불가 — 사람이 일일이 체크하면 빠뜨리는 것들을 기계가 잡아줘요.

**번들 사이즈도 체크할 수 있어요.**

```yaml
- name: 번들 사이즈 체크
  run: |
    SIZE=$(du -sk dist | cut -f1)
    if [ $SIZE -gt 500 ]; then
      echo "번들 사이즈 초과: ${SIZE}KB"
      exit 1
    fi
```

누군가 거대한 라이브러리를 무심코 import 해도 CI가 잡아줘요.

## 4단계: PR 리뷰 — 사람의 판단이 필요한 것

1~3단계를 통과한 코드만 리뷰 대상이 돼요. 이게 핵심이에요.

**리뷰어가 포맷팅이나 타입 에러를 지적할 필요가 없어요.** 그건 이미 기계가 다 잡았으니까요. 리뷰어는 기계가 못 잡는 것만 봐요.

- 이 컴포넌트 구조가 맞는가?
- 비즈니스 로직이 기획 의도와 일치하는가?
- 성능 이슈는 없는가?
- 더 나은 접근 방법이 있는가?

```markdown
<!-- PR 템플릿 -->
## 변경 사항
-

## 테스트
- [ ] 로컬 빌드 통과
- [ ] 키보드로 전체 기능 사용 가능
- [ ] 스크린 리더로 주요 흐름 확인

## 테스트하지 않은 것
-
```

OpenClaw 글에서 인상적이었던 건 **"무엇을 테스트하지 않았나"를 묻는 것**이에요. "다 테스트했습니다"보다 "이건 못 했습니다"가 리뷰어에게 훨씬 유용한 정보예요.

## 5단계: 배포 게이트 — 되돌릴 수 없는 행동 앞에서 멈추기

OpenClaw의 원칙: **"비가역적 행동 앞에서는 멈춘다."**

프론트엔드에서 비가역적 행동이 뭘까요?

- 프로덕션 배포
- npm 패키지 퍼블리시
- DNS / 리다이렉트 변경

이건 자동화하면 안 돼요. **사람이 확인하고 버튼을 눌러야 해요.**

```yaml
# 배포는 수동 트리거
on:
  workflow_dispatch:
    inputs:
      environment:
        description: '배포 환경'
        required: true
        type: choice
        options:
          - staging
          - production
```

HANUI에서는 npm 퍼블리시를 수동으로 해요. `changeset`으로 버전을 관리하고, 최종 퍼블리시는 사람이 확인 후 실행하는 방식이에요.

## 1인 프로젝트도 할 수 있다

"이건 대규모 프로젝트 얘기잖아" — 아니에요.

| 단계 | 대규모 프로젝트 | 1인 프로젝트 |
|------|----------------|-------------|
| 1. 커밋 전 | lint-staged + 경계 가드 | lint-staged |
| 2. 로컬 체크 | pnpm check (15개 스크립트) | pnpm build |
| 3. CI | 15개 병렬 잡 | GitHub Actions 기본 |
| 4. PR 리뷰 | CODEOWNERS + 팀 리뷰 | 셀프 리뷰 체크리스트 |
| 5. 배포 | 환경 승인 + 카나리 | 수동 배포 |

규모에 맞게 줄이면 돼요. 원칙은 같아요 — **싼 검사는 앞에, 비싼 검사는 뒤에.**

HANUI는 1인 프로젝트인데 1~3단계를 다 쓰고 있어요. 설정하는 데 반나절이면 되고, 한 번 세팅하면 매 커밋마다 알아서 돌아가요.

## 프론트엔드에서 특히 중요한 이유

백엔드는 타입 체크 + 테스트로 상당 부분 커버돼요. 프론트엔드는 그게 안 돼요.

- **시각적 회귀** — CSS 한 줄이 전체 레이아웃을 깨뜨릴 수 있어요
- **접근성** — aria 속성 하나 빠지면 스크린 리더 사용자가 막혀요
- **번들 사이즈** — `import moment from 'moment'` 한 줄이 300KB 추가해요
- **브라우저 호환성** — Chrome에서 되는데 Safari에서 안 되는 일이 비일비재해요

이걸 사람이 매번 체크하면 지쳐요. 게이트키핑은 **지치지 않는 검사관**을 세우는 거예요.

## 결론

OpenClaw가 보여준 건 "5단계 관문"이라는 구체적인 방법이 아니라, **"품질은 시스템으로 지킨다"**는 원칙이에요.

개발자의 의지력에 기대면 안 돼요. "꼼꼼히 보자"는 다짐은 3일이면 무너져요. 대신 기계가 잡을 수 있는 건 기계에게 맡기고, 사람은 기계가 못 하는 판단에 집중하는 거예요.

프론트엔드 프로젝트를 하고 있다면, 오늘 당장 1단계(lint-staged)부터 넣어보세요. 그것만으로도 코드 리뷰에서 포맷팅 지적이 사라져요. 그게 게이트키핑의 시작이에요.
