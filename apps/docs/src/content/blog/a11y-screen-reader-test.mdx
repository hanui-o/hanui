접근성 자동 검사 도구로 테스트하면 "문제 0건"이 나와요. 그런데 스크린 리더로 직접 써보면? 전혀 다른 세상이에요.

axe, Lighthouse 같은 자동 도구는 **전체 접근성 문제의 30% 정도**만 잡아요. 나머지 70%는 사람이 직접 써봐야 발견할 수 있는 문제예요. "버튼이라고 읽히긴 하는데 뭘 하는 버튼인지 모르겠다", "폼을 제출했는데 결과가 어디 있는지 모르겠다" 같은 것들이요.

오늘은 VoiceOver와 NVDA로 직접 앱을 테스트하는 방법을 공유할게요.

## 스크린 리더, 왜 직접 써봐야 하나

### 자동 검사 vs 수동 검사

```
자동 검사가 잡는 것 (30%):
✅ alt 텍스트 누락
✅ label 연결 안 됨
✅ 색상 대비 부족
✅ ARIA 속성 문법 오류
✅ 중복 id

자동 검사가 못 잡는 것 (70%):
❌ alt="이미지" 같은 무의미한 대체 텍스트
❌ 논리적 읽기 순서 문제
❌ 포커스 이동이 자연스러운지
❌ 동적 콘텐츠 업데이트 알림
❌ 커스텀 위젯의 실제 사용성
❌ 문맥 없는 "더보기" 링크
```

자동 도구가 "이미지에 alt가 있다"는 확인할 수 있지만, "이 alt가 사용자에게 의미 있는 정보를 전달하는가"는 판단할 수 없어요.

## VoiceOver 기본 조작 (Mac)

Mac을 쓰고 있다면 바로 테스트할 수 있어요. 별도 설치 없이 내장되어 있거든요.

### 시작과 종료

```
VoiceOver 켜기/끄기: Cmd + F5
(또는 Touch Bar 맥: Cmd + F5 3번 빠르게)

설정: 시스템 설정 → 손쉬운 사용 → VoiceOver
```

### 핵심 단축키

VoiceOver에서 `VO`는 **Control + Option** 키 조합이에요.

```
기본 탐색:
VO + →    다음 항목으로 이동
VO + ←    이전 항목으로 이동
VO + Space 활성화 (클릭)
Tab        다음 포커스 가능 요소
Shift+Tab  이전 포커스 가능 요소

빠른 탐색 (VO + U로 로터 열기):
VO + U     로터 열기 (헤딩, 링크, 폼 등 목록)
← →        로터 카테고리 전환
↑ ↓        항목 선택
Enter      해당 항목으로 이동

웹 탐색 (Quick Nav 켜기: ← + → 동시):
H          다음 헤딩
1~6        해당 레벨 헤딩
L          다음 링크
F          다음 폼 컨트롤
T          다음 테이블
```

### 처음 VoiceOver 켰을 때

처음 켜면 당황스러워요. 모든 걸 다 읽으니까요. 이렇게 시작해보세요:

```
1단계: Cmd + F5로 VoiceOver 켜기
2단계: VO + → 로 하나씩 이동해보기
3단계: 읽어주는 내용을 듣기
4단계: Tab으로 인터랙티브 요소만 이동해보기
5단계: 30분만 연습하면 기본은 됨
```

## NVDA 기본 조작 (Windows)

Windows에서는 NVDA를 추천해요. 무료이고, 한국 사용자 점유율도 높아요.

### 설치

[nvaccess.org](https://www.nvaccess.org/)에서 다운로드. 설치형과 포터블 모두 지원.

### 핵심 단축키

NVDA에서 `NVDA 키`는 기본적으로 **Insert** 또는 **Caps Lock**이에요.

```
기본 탐색:
NVDA + ↓   다음 줄 읽기
NVDA + ↑   이전 줄 읽기
Tab         다음 포커스 가능 요소
Shift+Tab   이전 포커스 가능 요소
Enter       활성화

빠른 탐색 (브라우저 모드):
H           다음 헤딩
1~6         해당 레벨 헤딩
K           다음 링크
F           다음 폼 필드
T           다음 테이블
D           다음 랜드마크
B           다음 버튼

모드 전환:
NVDA + Space   브라우저 모드 ↔ 포커스 모드
```

### 브라우저 모드 vs 포커스 모드

이게 NVDA의 가장 헷갈리는 부분이에요.

```
브라우저 모드 (기본):
- 페이지를 문서처럼 읽음
- H, K, F 같은 빠른 탐색 키 사용 가능
- 입력 필드에 타이핑 불가

포커스 모드:
- 특정 요소와 상호작용
- 입력 필드에 타이핑 가능
- 빠른 탐색 키 사용 불가 (그냥 타이핑됨)

자동 전환: 입력 필드에 Tab으로 이동하면 자동으로 포커스 모드 진입
수동 전환: NVDA + Space
```

## 실전 테스트 시나리오

이제 실제로 테스트해볼게요. 아래 시나리오대로 해보면 대부분의 문제를 발견할 수 있어요.

### 시나리오 1: 페이지 구조 확인

```
목표: 헤딩 구조가 논리적인지 확인

방법:
1. VoiceOver: VO + U → 헤딩 목록
   NVDA: NVDA + F7 → 헤딩 탭
2. 헤딩 목록만 보고 페이지 구조를 이해할 수 있는가?

✅ 좋은 구조:
h1: 이체
  h2: 출금 정보
  h2: 입금 정보
  h2: 이체 금액

❌ 나쁜 구조:
h1: (없음)
h3: 출금
h2: 입금
h5: 금액  ← 레벨이 뒤죽박죽
```

```tsx
// ❌ 스타일 때문에 헤딩 레벨을 바꿈
<h4 className="text-2xl font-bold">계좌 목록</h4>

// ✅ 올바른 레벨 + 스타일은 CSS로
<h2 className="text-lg">계좌 목록</h2>
```

헤딩 레벨은 **문서 구조**예요. 글자 크기를 바꾸고 싶으면 CSS를 쓰세요.

### 시나리오 2: 폼 입력 테스트

```
목표: 모든 입력 필드가 제대로 안내되는지 확인

방법:
1. Tab으로 각 입력 필드 이동
2. 스크린 리더가 읽어주는 내용 확인:
   - 필드 이름 (label)
   - 필드 타입 (텍스트, 콤보박스, 체크박스 등)
   - 필수 여부
   - 현재 값
   - 에러 메시지 (있다면)
```

```tsx
// ❌ 스크린 리더: "편집 텍스트" (뭘 입력하는지 모름)
<input type="text" placeholder="이름을 입력하세요" />

// ✅ 스크린 리더: "이름, 필수, 편집 텍스트"
<FormField required>
  <Label htmlFor="name">이름</Label>
  <Input id="name" type="text" aria-required="true" />
</FormField>
```

**placeholder는 label이 아니에요.** 입력을 시작하면 placeholder가 사라지면서 필드명을 잊게 되고, 스크린 리더에 따라 placeholder를 안 읽어주는 경우도 있어요.

### 시나리오 3: 동적 콘텐츠 알림

```
목표: 화면이 바뀔 때 스크린 리더에 알림이 가는지 확인

테스트 항목:
1. 폼 제출 후 성공/실패 메시지 → 읽히는가?
2. 검색 결과 업데이트 → "n개 결과" 안내가 되는가?
3. 토스트/알림 → 자동으로 읽히는가?
4. 로딩 상태 → "로딩 중" 안내가 되는가?
```

```tsx
// ❌ 검색 결과가 바뀌어도 스크린 리더는 모름
<div>{results.length}건의 검색 결과</div>

// ✅ 결과가 바뀌면 자동으로 읽어줌
<div aria-live="polite" aria-atomic="true">
  {results.length}건의 검색 결과
</div>

// ❌ 에러 메시지가 나타나도 스크린 리더는 모름
{error && <p className="text-red-500">{error}</p>}

// ✅ 에러 발생 시 즉시 알림
{error && <p role="alert">{error}</p>}
```

### 시나리오 4: 모달과 포커스

```
목표: 모달의 포커스 흐름이 올바른지 확인

체크리스트:
1. 모달 열기 → 포커스가 모달 안으로 이동하는가?
2. Tab으로 모달 밖을 벗어나지 않는가? (포커스 트랩)
3. Esc로 닫히는가?
4. 닫힌 후 포커스가 원래 위치(트리거)로 돌아가는가?
5. 모달 열릴 때 "대화상자" 역할이 안내되는가?
```

```tsx
// HANUI Modal은 이걸 전부 자동 처리
<Modal open={isOpen} onOpenChange={setIsOpen}>
  <ModalContent aria-labelledby="modal-title">
    <ModalHeader>
      <h2 id="modal-title">이체 확인</h2>
    </ModalHeader>
    <ModalBody>100,000원을 이체합니다.</ModalBody>
    <ModalFooter>
      <Button onClick={handleTransfer}>확인</Button>
      <Button variant="outline" onClick={() => setIsOpen(false)}>
        취소
      </Button>
    </ModalFooter>
  </ModalContent>
</Modal>
```

### 시나리오 5: 이미지와 아이콘

```
목표: 시각적 정보가 텍스트로 전달되는지 확인

체크리스트:
1. 의미 있는 이미지: alt가 내용을 설명하는가?
2. 장식용 이미지: alt=""로 스크린 리더가 무시하는가?
3. 아이콘 버튼: aria-label이 있는가?
4. 상태 표시 아이콘: 텍스트 대안이 있는가?
```

```tsx
// ❌ 스크린 리더: "버튼" (뭘 하는 버튼?)
<Button>
  <TrashIcon />
</Button>

// ✅ 스크린 리더: "삭제, 버튼"
<Button aria-label="삭제">
  <TrashIcon aria-hidden="true" />
</Button>

// ❌ 스크린 리더: "이미지, banner_2026_spring_sale_final_v3.png"
<img src="/banner.png" />

// ✅ 스크린 리더: "2026 봄 세일 — 전 상품 30% 할인"
<img src="/banner.png" alt="2026 봄 세일 — 전 상품 30% 할인" />

// ✅ 장식용 이미지는 숨기기
<img src="/divider.png" alt="" />
```

## 자주 발견되는 문제 TOP 10

실무에서 스크린 리더 테스트하면 반복적으로 나오는 문제들이에요.

| 순위 | 문제 | 증상 | 해결 |
|------|------|------|------|
| 1 | label 없는 입력 필드 | "편집 텍스트"로만 읽힘 | `<Label>` 연결 |
| 2 | `<div onClick>` 버튼 | Tab으로 도달 불가 | `<button>` 사용 |
| 3 | 무의미한 alt | "이미지", "사진" 등 | 내용 설명하는 alt |
| 4 | 모달 포커스 미관리 | 모달 뒤에 포커스 남음 | 포커스 트랩 구현 |
| 5 | 동적 업데이트 무알림 | 결과 바뀌어도 모름 | `aria-live` 사용 |
| 6 | "여기를 클릭하세요" 링크 | 링크 목록에서 구분 불가 | 목적지 설명하는 텍스트 |
| 7 | 헤딩 레벨 뒤죽박죽 | 구조 파악 불가 | 논리적 계층 유지 |
| 8 | `display: none` 콘텐츠 | 숨겼는데 읽힘/안 읽힘 | `aria-hidden` 적절 사용 |
| 9 | 커스텀 Select | Arrow 탐색 안 됨 | WAI-ARIA listbox 패턴 |
| 10 | 자동 재생 미디어 | 스크린 리더 음성을 덮음 | 자동 재생 금지 |

## 테스트 환경 세팅 팁

### 브라우저 조합

스크린 리더마다 궁합이 좋은 브라우저가 있어요.

```
VoiceOver + Safari   ← Mac 기본 조합 (가장 안정적)
NVDA + Firefox       ← Windows 추천 조합
NVDA + Chrome        ← 차선
JAWS + Chrome/Edge   ← 기업 환경 (유료)
```

### 음성 속도 조절

처음에는 느린 속도로 시작해요:

```
VoiceOver: VO + Cmd + ← / → (속도 조절)
NVDA: NVDA + Ctrl + ↑ / ↓ (속도 조절)
```

익숙해지면 속도를 올려보세요. 실제 스크린 리더 사용자는 2~3배속으로 듣는 경우가 많아요.

### 결과 기록 템플릿

```markdown
## 접근성 테스트 결과

**테스트 환경**: VoiceOver + Safari 18 / macOS
**테스트 일시**: 2026-03-20
**테스트 페이지**: /transfer (이체 페이지)

### 발견 사항

| # | 심각도 | 위치 | 문제 | 예상 동작 | 실제 동작 |
|---|--------|------|------|----------|----------|
| 1 | 높음 | 금액 입력 | label 없음 | "이체 금액, 편집" | "편집 텍스트" |
| 2 | 중간 | 확인 모달 | 포커스 미이동 | 모달 내부 포커스 | 모달 뒤에 포커스 |
| 3 | 낮음 | 결과 페이지 | 결과 미알림 | "이체 완료" 자동 읽기 | 무음 |
```

이런 식으로 기록하면 개발자가 바로 수정할 수 있어요.

## 마무리

스크린 리더 테스트는 처음에 어색하지만, 30분만 투자하면 기본은 할 수 있어요. 그리고 한 번이라도 해보면 "아, 이래서 접근성이 중요하구나"가 체감이 돼요.

자동 검사 도구를 돌려서 "문제 0건"이 나오더라도, 스크린 리더로 한 번만 더 확인해보세요. 진짜 문제는 거기서 나와요.

다음 편에서는 이런 테스트 과정을 자동화하는 방법을 다뤄볼게요. axe + Storybook + CI 파이프라인으로 접근성을 디자인 시스템 레벨에서 관리하는 방법이에요.
