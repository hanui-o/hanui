접근성 테스트, 배포 직전에 몰아서 하고 계신가요?

그러면 이미 늦어요. 코드가 쌓인 뒤에 접근성 문제를 고치는 건 — 렌더 이후에 레이아웃 다시 짜는 것만큼 힘들어요.

해답은 **"처음부터 자동으로 막는 것"**이에요. 컴포넌트 작성할 때, PR 올릴 때, 배포 전에. 각 단계마다 자동 검사가 붙으면 접근성 문제가 운영에 나갈 일이 없어요.

오늘은 HANUI 디자인 시스템에 적용한 접근성 자동화 파이프라인을 공유할게요.

## 3단계 자동화 파이프라인

```
컴포넌트 작성 → 단위 테스트 (axe-core + vitest)
      ↓
Storybook 확인 → 시각 검사 (a11y 애드온)
      ↓
PR 올리기 → CI 자동 차단 (GitHub Actions)
```

각 단계마다 다른 종류의 문제를 잡아요. 겹치는 게 있어도 괜찮아요 — 그게 안전망이에요.

## 1단계: axe-core 단위 테스트

### 설치

```bash
pnpm add -D vitest-axe @axe-core/react axe-core
```

### 기본 패턴

```ts
// button.test.tsx
import { render } from '@testing-library/react';
import { axe, toHaveNoViolations } from 'vitest-axe';
import { expect } from 'vitest';
import { Button } from './button';

expect.extend(toHaveNoViolations);

describe('Button 접근성', () => {
  it('기본 버튼은 접근성 위반이 없어야 한다', async () => {
    const { container } = render(<Button>저장하기</Button>);
    const results = await axe(container);
    expect(results).toHaveNoViolations();
  });

  it('비활성 버튼도 접근성을 충족해야 한다', async () => {
    const { container } = render(<Button disabled>저장하기</Button>);
    const results = await axe(container);
    expect(results).toHaveNoViolations();
  });

  it('아이콘만 있는 버튼은 aria-label이 필요하다', async () => {
    const { container } = render(
      <Button aria-label="닫기">
        <CloseIcon />
      </Button>
    );
    const results = await axe(container);
    expect(results).toHaveNoViolations();
  });
});
```

```ts

export default {
  test: {
    coverage: {
      reporter: ['text', 'lcov'],
    },
  },

```

테스트 커버리지랑 접근성 검사를 같이 돌리면 어떤 컴포넌트가 빠졌는지 한눈에 보여요.

## 2단계: Storybook a11y 애드온

axe 테스트는 코드 레벨에서 잡는 거예요. 근데 **"실제로 어떻게 보이는가"**는 Storybook에서 봐야 해요.

### 설치

```bash
pnpm add -D @storybook/addon-a11y
```

```ts
// .storybook/main.ts
export default {
  addons: [
    '@storybook/addon-a11y',
    // ... 다른 애드온들
  ],
};
```

### 스토리별 접근성 설정

```tsx
// button.stories.tsx
import type { Meta, StoryObj } from '@storybook/react';
import { Button } from './button';

const meta: Meta<typeof Button> = {
  component: Button,
  parameters: {
    a11y: {
      // 이 스토리만 특정 규칙 제외
      config: {
        rules: [
          {
            id: 'color-contrast',
            enabled: false, // 브랜드 색상 사용 시 예외 처리
          },
        ],
      },
    },
  },
};

export default meta;

export const Default: StoryObj = {
  args: { children: '저장하기' },
};

export const Disabled: StoryObj = {
  args: { children: '저장하기', disabled: true },
};

// 접근성 위반이 예상되는 케이스 — 문서화 목적
export const WithoutLabel: StoryObj = {
  args: { children: <CloseIcon /> },
  parameters: {
    a11y: { disable: true }, // 의도적으로 검사 끔
  },
};
```

### Storybook에서 보이는 것들

a11y 탭에서 실시간으로 보여요:

- **위반 (Violations)**: 반드시 고쳐야 하는 것
- **불완전 (Incomplete)**: 수동 확인 필요
- **통과 (Passes)**: 문제 없는 것

색상 대비 문제는 axe 테스트에서는 잘 안 잡히는데 Storybook에서 시각적으로 확인하기 좋아요.

## 3단계: CI 자동 차단

로컬에서 테스트 다 통과해도 PR에서 놓치는 경우가 있어요. CI에 붙이면 완전히 막을 수 있어요.

### GitHub Actions 설정

```yaml
# .github/workflows/a11y.yml
name: 접근성 검사

on:
  pull_request:
    paths:
      - 'packages/react/src/**'
      - 'apps/docs/src/**'

jobs:
  a11y-test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - name: pnpm 설치
        uses: pnpm/action-setup@v3
        with:
          version: 9

      - name: Node.js 설정
        uses: actions/setup-node@v4
        with:
          node-version: 22
          cache: 'pnpm'

      - name: 의존성 설치
        run: pnpm install

      - name: 접근성 단위 테스트
        run: pnpm test --reporter=verbose

      - name: Storybook 빌드
        run: pnpm --filter docs build-storybook

      - name: Storybook 접근성 테스트
        run: pnpm --filter docs test-storybook
```

### Storybook 자동 테스트 설정

```bash
pnpm add -D @storybook/test-runner
```

```json
// package.json (docs)
{
  "scripts": {
    "test-storybook": "test-storybook --url http://localhost:6006"
  }
}
```

```ts
// .storybook/test-runner.ts
import { checkA11y, injectAxe } from 'axe-playwright';

module.exports = {
  async preVisit(page) {
    await injectAxe(page);
  },
  async postVisit(page) {
    await checkA11y(page, '#storybook-root', {
      detailedReport: true,
      detailedReportOptions: {
        html: true,
      },
    });
  },
};
```

이렇게 하면 **모든 Story를 자동으로 접근성 검사**해요. 새 컴포넌트 추가할 때 Story만 작성하면 자동으로 검사 대상이 돼요.

## HANUI 파이프라인 전체 그림

```
개발자가 컴포넌트 작성
      ↓
vitest + axe 테스트 (로컬)
      ↓
Storybook a11y 탭 확인 (로컬)
      ↓
PR 생성
      ↓
GitHub Actions
  ├─ pnpm test (axe 단위 테스트)
  └─ test-storybook (전체 Story 접근성 검사)
      ↓
모두 통과 → 머지 가능
실패 → 머지 차단
```

실제로 이 파이프라인 붙이고 나서 **운영에 접근성 문제가 나간 게 0건**이에요. 개발 중에 다 잡히거든요.

## 자동화로 못 잡는 것들

자동화가 완벽하진 않아요. 놓치는 부분도 알고 있어야 해요.

| 자동 검사 가능 | 수동 검사 필요 |
|--------------|--------------|
| alt 텍스트 누락 | alt 내용이 의미 있는지 |
| label 연결 | 읽기 순서가 자연스러운지 |
| 색상 대비 비율 | 실제 스크린 리더 경험 |
| ARIA 문법 | 인지 부하가 적절한지 |
| 키보드 포커스 표시 | 실제 키보드 사용 흐름 |

4편에서 다룬 VoiceOver/NVDA 테스트는 자동화랑 병행해야 해요.

## 접근성 점수 추적

GitHub Actions에서 배지도 달 수 있어요:

```yaml
- name: 접근성 결과 저장
  if: always()
  uses: actions/upload-artifact@v4
  with:
    name: a11y-report
    path: a11y-report.json
```

PR 코멘트로 결과 자동 게시:

```yaml
- name: PR 코멘트
  uses: marocchino/sticky-pull-request-comment@v2
  with:
    message: |
      ## 접근성 검사 결과
      ✅ 위반: 0건
      ⚠️ 불완전: 2건 (수동 확인 필요)
      ✅ 통과: 47건
```
