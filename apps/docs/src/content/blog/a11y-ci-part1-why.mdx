접근성 CI 구축기 1편이에요.

최근에 운영 중인 서비스에 **웹 접근성 자동 검사 파이프라인**을 붙이는 프로젝트를 주도하고 있어요. 코드보다 먼저 **"왜 이게 필요했는지"**를 정리해두고 싶어서 시리즈로 기록을 시작해요.

오늘은 1편. 배경과 결정 과정이에요. 구체적인 코드는 2편부터 다룰게요.

## 접근성 검사, 지금까지 어떻게 해왔나

대부분의 팀은 이렇게 해요:

1. QA 팀이 체크리스트 들고 수동 테스트
2. 스크린 리더(NVDA, VoiceOver)로 몇 개 페이지 훑기
3. 디자인 QA에서 색상 대비 눈으로 확인
4. 접근성 컨설팅 업체에 한 번씩 의뢰

문제는 **이 모든 게 "이벤트성"**이라는 거예요. 릴리스 전에 한 번, 감사 때 한 번, 컨설팅 맡길 때 한 번. 그 사이에 새로 들어간 코드는 아무도 안 봐요.

## 수동 검사의 한계 3가지

### 1. 회귀를 못 잡아요

버그 고쳤는데 다른 곳이 깨지는 건 개발자 누구나 아는 거예요. 접근성도 똑같아요.

예를 들어:

- 디자이너가 버튼 색상을 바꿨는데 대비율이 4.5:1 밑으로 떨어짐
- 개발자가 모달 리팩토링하면서 포커스 트랩 로직을 빼먹음
- 번역팀이 한국어 텍스트를 추가했는데 `lang` 속성 업데이트 안 됨

전부 **한 번 통과했던 것들**이에요. 수동으로는 매번 전 페이지 재검사 불가능해요.

### 2. 커버리지가 너무 낮아요

QA가 접근성 검사하는 페이지는 보통 로그인, 홈, 주요 대시보드 정도. 관리자 페이지, 설정 하위 메뉴, 에러 페이지 등은 **거의 안 봐요**.

근데 접근성 인증(VPAT, KWCAG) 받으려면 **모든 페이지**가 대상이에요. 특정 페이지만 통과하면 의미가 없어요.

### 3. 사람마다 기준이 달라요

같은 WCAG 2.1 AA 기준을 봐도:

- A씨는 "alt가 있으면 통과"로 판단
- B씨는 "alt가 의미 있게 쓰여야 통과"로 판단
- C씨는 "장식 이미지면 alt=''여야 엄격히 통과"로 판단

axe-core 같은 엔진은 이걸 **기계적 기준**으로 통일해줘요. 사람의 판단이 필요한 영역은 따로 남기되, 적어도 "컴파일 타임에 잡히는 위반"은 자동화할 수 있어요.

## VPAT가 뭐고 왜 중요한가

**VPAT (Voluntary Product Accessibility Template)**는 제품의 접근성 준수 상태를 기록한 문서예요. 미국에서 시작됐지만 지금은 글로벌 표준처럼 쓰여요.

VPAT가 왜 중요한가:

| 대상 | 왜 요구하는가 |
|------|--------------|
| **미국 연방 정부 조달** | Section 508 준수 증빙 — 사실상 필수 |
| **EU 기업 조달** | EN 301 549 준수 증빙 |
| **글로벌 대기업** | 자사 구매 정책상 공급사 검증 요건 |
| **한국 공공기관** | KWCAG 2.2 — VPAT는 아니지만 유사한 준수 문서 필요 |

제품을 팔거나 B2B 계약을 따려면 VPAT 비슷한 문서를 **증빙**해야 하는 시대예요. 그리고 그 증빙이 **근거 있어야** 해요.

"저희가 수동 검사 한 번 했어요"는 더 이상 통하지 않아요. **지속적으로 모니터링한 증거**가 필요해요. 그게 CI 파이프라인이에요.

## CI가 해결하는 것 / 해결 못 하는 것

기대를 정확히 설정하는 게 중요해요. 자동 검사는 **만병통치약이 아니에요.**

### CI가 해결하는 것 (axe-core 기준)

- 색상 대비 (contrast)
- ARIA 속성 누락/오용
- form label 연결
- heading 구조
- landmark 누락
- alt 속성 누락
- 중복 ID
- 키보드 포커스 가능 여부 (요소 단위)

**대략 WCAG 위반의 30~40%를 자동으로 잡아요.** 생각보다 적죠.

### CI가 해결 못 하는 것

- alt 텍스트가 **의미 있는지** (alt="image" 같은 건 통과해버림)
- 키보드 네비게이션의 **논리적 순서**
- 스크린 리더 **실제 음성 흐름**
- 포커스가 **보이는지** (기술적으로만 확인)
- 동영상 자막 **품질**
- 에러 메시지의 **명확성**

이건 사람이 테스트해야 해요. 자동 검사는 이 영역을 대체하는 게 아니라, **사람이 더 깊은 검사에 집중할 수 있게 바닥을 깔아주는 거예요.**

## 조직에 어떻게 설득했나

프론트엔드 팀 혼자 "접근성 CI 붙여요"라고 하면 대부분 안 돌아가요. 다른 팀의 리소스가 필요하거든요:

- 인프라팀: CI 러너 리소스
- 보안팀: 테스트용 계정 크리덴셜 관리
- QA팀: 수동 검사와 역할 분담
- 디자인팀: 색상 대비 기준 합의

설득할 때 효과적이었던 프레이밍은 3가지였어요.

### 1. "이건 품질 문제가 아니라 법/계약 리스크예요"

미국에서 ADA 소송으로 기업이 패소한 사례, EU에서 접근성 미준수로 서비스 제한된 사례를 보여줬어요. 한국도 장애인차별금지법이 있어서 이론상 소송 가능한 영역이에요.

**"품질 개선"**이 아니라 **"리스크 관리"**로 프레이밍하면 경영진이 달리 들어요.

### 2. "수동 검사 비용을 계산해보죠"

매 릴리스마다 QA가 2~3일 접근성 체크리스트 돌리면 **인건비 X N번**이에요. CI로 자동화하면 초기 구축 비용은 있지만 **운영 비용은 거의 0**이에요.

엑셀로 "이번 분기 접근성 QA 공수 vs CI 도입 시 절감"을 그려보면 설득이 빨라요.

### 3. "한 번에 다 고치자가 아니라 회귀만 막자"

가장 중요한 포인트예요. 기존 코드에 이미 위반이 있는 상태에서 "위반 0 강제"를 걸면 **아무도 PR을 못 머지해요.**

그래서 **"더 늘지만 마"** 정책을 제안했어요:

- 현재 위반 수를 베이스라인으로 기록
- PR로 위반이 **늘어나면 차단**
- 기존 위반은 별도 개선 스프린트로 줄여나감

이 단계별 접근을 보여줘야 팀이 받아들여요. "다 고쳐야 한다"는 부담을 지우면 거부감이 엄청 커져요.

## 초기 결정 포인트 5가지

프로젝트 킥오프할 때 정해야 할 것들을 정리해둘게요. 뒤에서 다시 결정하면 비용이 커요.

### ① 기준을 어디로 잡을까

- **WCAG 2.1 AA**가 현실적 기준 (가장 많이 쓰임)
- AAA는 너무 빡세서 실무에서 거의 안 씀
- 한국은 **KWCAG 2.2**, 대부분 WCAG 2.1 AA와 호환

**결정: WCAG 2.1 AA**

### ② 컴포넌트 단위 vs 페이지 단위 vs 둘 다

- **컴포넌트 단위**: Storybook + axe로 Button/Input 등 검증 — 빠름, but 조합 이슈 못 잡음
- **페이지 단위**: Playwright + axe로 실제 페이지 스캔 — 느리지만 실제 사용자 경험 반영
- **둘 다**: 이상적이지만 구축/유지비용 2배

**결정: 페이지 단위 우선. 디자인 시스템 자체는 이미 단위 테스트 있으니 중복 회피.**

### ③ 인증된 페이지까지 스캔할까

- 비로그인 페이지만 보면 **범위가 너무 작아짐**
- 인증 넘으려면 storage state 관리, 크리덴셜 보안 등 복잡도 증가
- 하지만 실제 서비스의 핵심은 대부분 인증 뒤에 있음

**결정: 인증 페이지 포함. 복잡도 감수.**

### ④ 위반 발견 시 CI를 실패시킬까

- 처음부터 실패 → 기존 위반에 막혀서 도입 불가
- 실패 안 시킴 → 팀이 "통과했네" 하고 지나감
- **타협: PR 코멘트로 수치만 표시하는 베이스라인 단계를 거쳐서 게이팅으로 전환**

**결정: Phase 1(베이스라인, 2~4주) → Phase 2(게이팅, 증가 시 차단).**

### ⑤ 누가 결과를 보고 대응할까

- 자동 검사만 만들어놓고 **아무도 안 보면 의미 없음**
- PR 코멘트로 리뷰어가 자연스럽게 보게 하는 게 핵심
- 주간 회고에 "이번 주 a11y 위반 증감"을 고정 항목으로 넣는 것도 좋음

**결정: PR 리뷰 프로세스에 통합. 월 1회 트렌드 리포트 공유.**

## 이 시리즈에서 다룰 내용

앞으로 편별 주제 미리 공유할게요:

- **1편 (이번 글)**: 왜 만들기로 했나 — VPAT, 수동 검사 한계, 조직 설득
- **2편**: Playwright + axe-core 파이프라인 구축 — 실제 코드와 GitHub Actions 설정
- **3편**: CI 결과 제대로 읽는 법 — 베이스라인, 초록 체크의 함정, 스캔 실패 구별
- **4편**: credential 보안 딜레마 — storage state 관리, 토큰 만료, 해킹 대비

다음 편에서 실제 **Playwright + axe-core로 어떻게 CI를 구축했는지** 코드 포함해서 다룰게요.
