"디자인 나오면 작업할게요."

프론트엔드 개발자라면 한 번쯤 이 말을 해봤거나, 들어봤을 거예요.
스프린트 계획 회의에서 디자이너가 "이번 주까지 시안 나와요"라고 하면, 개발자는 자연스럽게 다음 주부터 작업을 잡아요. 그리고 디자인이 하루 늦어지면, 개발도 하루 밀리고, QA도 밀리고, 릴리즈도 밀려요.

20년간 프로젝트를 하면서 이 패턴을 수없이 봤어요. 그리고 확실하게 말할 수 있는 건 — **이건 애자일이 아니라 워터폴이에요.**

## 디자인 완료를 기다리는 건 워터폴이다

워터폴의 핵심은 **이전 단계가 완료되어야 다음 단계를 시작**하는 거예요.

```
기획 완료 → 디자인 완료 → 개발 시작 → QA → 배포
```

이걸 2주 스프린트에 욱여넣으면 어떻게 될까요?

- 1주차: 디자인 진행 중, 개발자 대기
- 2주차 초반: 디자인 전달, 개발 시작
- 2주차 후반: 개발 미완료, QA 불가

**2주 스프린트인데 실제 개발 기간은 3~4일.** 이래서 항상 일정이 부족하다고 느끼는 거예요.

## 와이어프레임만 있으면 시작할 수 있는 것들

디자인이 "완성"되기 전에도 할 수 있는 일이 놀라울 정도로 많아요.

### 1. 레이아웃과 라우팅

와이어프레임에는 페이지 구조가 다 나와 있어요. 헤더, 사이드바, 메인 콘텐츠 영역, 푸터 — 이건 색상이나 폰트와 무관해요.

```tsx
// 와이어프레임만 보고 바로 만들 수 있는 것
export default function DashboardLayout({ children }) {
  return (
    <AdminLayout>
      <SideNavigation items={menuItems} />
      <main>{children}</main>
    </AdminLayout>
  );
}
```

### 2. 컴포넌트 구조와 Props 설계

"사용자 목록 테이블이 있고, 검색과 필터가 있다" — 이 정보만으로 충분해요.

```tsx
interface UserTableProps {
  users: User[];
  onSearch: (query: string) => void;
  onFilter: (status: UserStatus) => void;
  onSort: (column: string, direction: 'asc' | 'desc') => void;
}
```

색상이 파란색이든 빨간색이든, 이 구조는 바뀌지 않아요.

### 3. API 연동과 상태 관리

데이터를 불러오고, 에러를 처리하고, 로딩 상태를 관리하는 건 디자인과 무관해요.

```tsx
const { data: users, isLoading, error } = useQuery({
  queryKey: ['users', filters],
  queryFn: () => fetchUsers(filters),
});
```

### 4. 폼 로직과 유효성 검증

"회원가입 폼에 이메일, 비밀번호, 이름이 필요하다" — 와이어프레임에 다 있어요.

```tsx
const schema = z.object({
  email: z.string().email('올바른 이메일을 입력하세요'),
  password: z.string().min(8, '8자 이상 입력하세요'),
  name: z.string().min(2, '이름을 입력하세요'),
});
```

버튼이 둥근지 네모인지는 나중에 입히면 돼요.

### 5. 접근성과 키보드 네비게이션

포커스 순서, aria 속성, 키보드 트랩 방지 — 이건 시각적 디자인이 아니라 **인터랙션 설계**예요. 와이어프레임 단계에서 오히려 더 잘 보여요.

## 디자인 토큰이 있으면 색상도 나중에 입힌다

디자인 시스템을 쓰고 있다면, **디자인 완성 전에도 실제와 거의 동일한 UI**를 만들 수 있어요.

```css
/* 디자인 토큰 — 값은 나중에 바뀔 수 있지만 구조는 유지 */
--color-primary: #0066FF;
--color-surface: #FFFFFF;
--color-text: #171717;
--spacing-md: 16px;
--radius-md: 8px;
```

HANUI를 만들면서 이걸 직접 경험했어요. KRDS 디자인 토큰으로 컴포넌트를 만들어놓으면, 나중에 토큰 값만 바꿔도 전체 UI가 한 번에 바뀌어요. 개별 컴포넌트를 하나씩 수정할 필요가 없죠.

```tsx
// 디자인이 확정되기 전에도 이렇게 만들 수 있어요
<Button variant="primary">저장</Button>
<Card>
  <CardHeader>사용자 정보</CardHeader>
  <CardContent>...</CardContent>
</Card>
```

`primary`가 파란색이든 초록색이든, 코드는 그대로예요.

## 실제 스프린트에서의 타임라인

디자인을 기다리지 않는 스프린트는 이렇게 돌아가요.

| 일차 | 디자이너 | UI 개발자 |
|------|----------|-----------|
| Day 1 | 와이어프레임 공유 | 레이아웃, 라우팅 구성 |
| Day 2 | 시각 디자인 시작 | API 연동, 상태 관리 |
| Day 3 | 시각 디자인 진행 | 컴포넌트 구조, 폼 로직 |
| Day 4 | 시각 디자인 완료 | 접근성, 키보드 네비게이션 |
| Day 5 | 디자인 QA | 디자인 토큰 적용, 스타일 미세 조정 |
| Day 6~7 | 수정 사항 반영 | 수정 반영 + QA |

**핵심은 Day 5에요.** 디자인이 확정되면 그때 색상, 간격, 폰트를 입혀요. 구조는 이미 완성되어 있으니까 하루면 충분해요.

기존 방식이면 Day 4부터 시작해서 7일 안에 끝내야 하는데 — 이 방식이면 5일 동안 충분히 작업한 상태에서 마무리만 하면 돼요.

## 이걸 팀에 어떻게 설득하나

솔직히 이게 가장 어려운 부분이에요. "디자인 없이 어떻게 개발해요?"라는 반응이 자연스러우니까요.

### PM에게

> "개발 리드타임을 줄이면 스프린트 완료율이 올라갑니다. 디자인과 개발을 병렬로 진행하면 2주 스프린트에서 실제 개발 가능 일수가 3일에서 7일로 늘어나요."

숫자로 말하면 통해요.

### 디자이너에게

> "와이어프레임 단계에서 같이 보면, '이건 구현이 어려워요'를 미리 말씀드릴 수 있어요. 시안 다 만든 다음에 뒤집는 것보다 훨씬 효율적이에요."

디자이너 입장에서도 나중에 "이건 안 돼요"를 듣는 것보다, 초기에 피드백 받는 게 훨씬 낫거든요.

### 다른 개발자에게

> "컴포넌트 구조는 디자인이 바뀌어도 안 바뀌어요. 색상이 파란색에서 초록색으로 바뀐다고 `useQuery` 훅을 다시 짜나요?"

이건 한 번만 경험하면 자연스럽게 체득돼요.

## 전제 조건: 디자인 시스템

이 모든 게 가능하려면 **디자인 시스템**이 있어야 해요. 디자인 토큰 없이 와이어프레임 단계에서 개발을 시작하면, 디자인 확정 후 전부 다시 짜야 하는 상황이 벌어질 수 있어요.

디자인 시스템이 해주는 역할:

- **토큰**: 색상, 간격, 타이포 등의 값을 변수로 관리 → 나중에 값만 바꾸면 됨
- **컴포넌트**: Button, Input, Card 같은 기본 블록이 이미 있음 → 조립만 하면 됨
- **패턴**: 폼, 테이블, 모달 같은 반복 패턴이 정해져 있음 → 매번 새로 설계 안 해도 됨

HANUI를 만든 이유 중 하나가 바로 이거였어요. 프로젝트마다 컴포넌트를 새로 만들다 보면, 디자인이 나와야만 개발을 시작할 수 있는 구조가 돼버리거든요.

## 마무리

"디자인 나오면 작업할게요"는 편한 말이에요. 기다리면 되니까요.

하지만 애자일에서 UI 개발자의 가치는 **기다리지 않는 것**에 있어요. 와이어프레임 단계에서 구조를 잡고, API를 연동하고, 접근성을 챙기고 — 디자인이 확정되면 옷만 입히면 되는 상태를 만들어놓는 거예요.

이렇게 하면:
- 스프린트 완료율이 올라가고
- 디자인 변경에 흔들리지 않고
- "프론트엔드가 병목이에요"라는 말을 안 듣게 돼요

20년간 배운 건 — **시니어 UI 개발자는 디자인을 기다리지 않는다**는 거예요.
