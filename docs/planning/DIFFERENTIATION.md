## 1. 현황 분석

### 1.1 기존 KRDS 라이브러리의 공통 약점

```
모두 "컴포넌트 라이브러리"에만 집중

❌ 프로젝트 시작을 도와주지 않음
   → 개발자가 직접 세팅 (Webpack, Vite, ESLint...)

❌ 실무 템플릿 부족
   → "Button 컴포넌트는 있는데, 로그인 페이지는?"

❌ 접근성 검증을 개발자에게 맡김
   → 수동 체크 → 시간 낭비 → 검수에서 지적

❌ 문서가 딱딱함
   → 기술 문서 수준
   → "이거 써서 뭘 만들 수 있는지" 감이 안 옴

❌ 생태계 없음
   → 일회성 라이브러리
   → 커뮤니티 활동 미미
```

### 1.2 우리의 기회

```
✅ 시장 검증 완료
   - 여러 개발자가 각자 만듦 = 수요 있음
   - 하지만 "표준"은 없음

✅ 진입장벽 낮음
   - Star 50개 수준 = 아직 메이저 없음
   - 품질로 역전 가능

✅ 공공 SI 시장 거대
   - 연 9,000억원
   - KRDS 의무 사용
   - 모든 공공 프로젝트가 잠재 고객
```

## 2. 4가지 차별화 전략

### 전략 ① "실무형 KRDS Starter Kit"

#### 기존 방식:
```
1. 개발자가 Vite/Webpack 세팅
2. React/Vue 설치
3. KRDS 라이브러리 찾기
4. npm install @krds/something
5. 컴포넌트 하나씩 import
6. 페이지 직접 구성
7. Storybook 세팅 (선택)
8. ESLint 설정
9. 접근성 검사 도구 추가
10. ...

→ 총 1~2주 소요
```

#### KRDS Platform 방식:
```bash
npx create-krds-app my-project

? Framework? › React
? Include templates? › Yes

✓ 프로젝트 생성 완료! (30초)

cd my-project
npm run dev

→ 이미 작동하는 로그인/대시보드/게시판 포함
→ Storybook, ESLint, 접근성 검사 모두 설정 완료
```

#### 핵심 차별점:
```
단순 컴포넌트 라이브러리 ❌
프로젝트 템플릿 세트 ✅

포함 내용:
├── KRDS 컴포넌트 (커스터마이징 가능)
├── 페이지 템플릿 3개+
├── Storybook 설정
├── ESLint (KRDS 규칙)
├── 접근성 검사 설정
├── 타입스크립트
├── Git 초기화
└── README (프로젝트별 가이드)

→ "바로 개발 시작 가능"
```

#### 벤치마크:
```
Create React App: 기본 세팅만
Next.js: SSR 특화
KRDS Platform: 공공 프로젝트 특화 ⭐
```

### 전략 ② "KRDS + 접근성 자동 검수 도구 연계"

#### 현재 문제:
```
공공 프로젝트 필수: 한국형 웹 접근성 인증

과정:
1. 개발 완료
2. 접근성 전문가에게 검수 의뢰
3. 지적사항 받음 (20~50개)
4. 개발자가 수정
5. 재검수
6. 반복...

→ 2~4주 소요
→ 비용 500만원~1000만원
→ 일정 지연
```

#### KRDS Platform 해결책:

**A. VSCode Extension "KRDS Validator"**
```typescript
// 개발 중 실시간 검사

<Button>클릭</Button>
     ↑
     ⚠️ Warning: aria-label이 없습니다

💡 Quick Fix:
<Button aria-label="클릭 버튼">클릭</Button>
```

**실시간 검사 항목:**
```
✓ ARIA 속성 (role, label, describedby...)
✓ 키보드 접근성 (tabIndex, onKeyPress...)
✓ 색상 대비 (WCAG AA 이상)
✓ 초점 표시 (focus-visible)
✓ 대체 텍스트 (alt, title)
✓ 폼 레이블 (label for input)
✓ 헤딩 구조 (h1, h2...)
```

**자동 수정 기능:**
```
Ctrl+Shift+P
→ "KRDS: Fix All Accessibility Issues"

변경 사항:
✓ aria-label 자동 추가
✓ alt 속성 생성 (AI)
✓ role 속성 추가
✓ tabIndex 최적화

→ 3분 안에 모든 접근성 이슈 해결
```

**B. CI/CD 통합**
```yaml
# GitHub Actions
name: Accessibility Check

on: [pull_request]

jobs:
  a11y:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v3
      - name: KRDS A11y Check
        run: npx @krds/cli check
      - name: Comment PR
        if: failure()
        run: |
          # PR에 접근성 위반 사항 자동 코멘트
```

**결과:**
```
PR 올리면 자동 검사
→ 접근성 통과 못하면 머지 블락
→ 개발자는 배포 전에 모든 이슈 해결

최종 검수 시:
→ 지적사항 0개
→ 검수 기간 1일
→ 비용 50만원 (형식적 검증만)

시간 절약: 2~4주 → 1일
비용 절약: 500만원 → 50만원
```

#### 경쟁 우위:
```
기존 도구 (Lighthouse, axe-core):
- 범용적 검사 (글로벌 기준)
- 한국 웹 접근성 기준 미흡
- 개발자가 해석/수정

KRDS Validator:
- 한국형 웹 접근성 특화 ⭐
- 자동 수정 기능 ⭐
- KRDS 가이드 연동 ⭐
- 실시간 피드백 ⭐
```

```
"이미 있다"는 이유로 포기할 필요 없음

```
KRDS Platform =
  컴포넌트 라이브러리 (기본)
