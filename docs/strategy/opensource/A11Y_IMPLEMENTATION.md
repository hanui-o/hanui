## Q1. 어떤 접근성 요소부터 구현할까?

> **핵심 원칙**: 접근성을 가장 빠르게 체감하고, 개발자·디자이너 모두에게 "바로 효용이 보이는" 요소를 먼저 자동화

### 우선순위 4대 요소

---

### 1) 초점(Focus) 관리 자동화 ⭐ 최우선

**가장 강력하고 즉각적인 효과**를 제공합니다. Radix UI가 처음에 극찬받은 이유 중 하나입니다.

#### 자동화 포인트

| 요소                                | 설명                                            | 효과                        |
| ----------------------------------- | ----------------------------------------------- | --------------------------- |
| **포커스 트랩** (Focus Trap)        | 모달/다이얼로그에서 포커스가 밖으로 나가지 않음 | 즉각적인 UX 개선            |
| **포커스 복원** (Focus Restoration) | 모달 닫힌 후 원래 위치로 포커스 복귀            | 키보드 사용자 편의성 극대화 |
| **키보드 네비게이션**               | Tab, Shift+Tab, Arrow keys 자동 처리            | 개발 시간 절약              |
| **포커스 인디케이터**               | 기본 포커스 링 스타일 제공                      | 시각적 일관성               |

#### 즉각적 효과

- 사용자가 바로 체감
- 개발자 시간 절약
- **WCAG 2.2의 Keyboard Accessible (2.1.1)** 영역 직결

**참고**: [WCAG 2.2 Guidelines](https://www.w3.org/WAI/WCAG22/)

---

### 2) ARIA role + 속성 자동 삽입

**컴포넌트 사용 시 자동으로 접근성 속성이 들어가도록** 만드는 것이 핵심입니다.

#### 자동화 예시

```tsx
// 개발자가 작성하는 코드
<Button>클릭하세요</Button>

// HANUI가 자동으로 생성하는 HTML
<button
  role="button"
  aria-disabled="false"
  type="button"
>
  클릭하세요
</button>
```

#### 주요 컴포넌트별 자동 ARIA

| 컴포넌트      | 자동 적용 ARIA                              |
| ------------- | ------------------------------------------- |
| `<Button>`    | `role="button"` + disabled 상태 ARIA 처리   |
| `<Modal>`     | `role="dialog"` + `aria-modal="true"`       |
| `<Tabs>`      | `role="tablist"`, `aria-selected` 자동 처리 |
| `<Accordion>` | `aria-expanded`, `aria-controls`            |
| `<Menu>`      | `role="menu"`, `aria-haspopup`              |

#### 핵심 가치

> 대부분의 개발자들이 "정확한 ARIA를 넣는 것"을 힘들어하므로 자동화 가치가 매우 큽니다.

---

### 3) 필수 연관 요소 자동 연결

**특히 Form·Input 계열에서 효과가 큽니다.**

#### 자동화 포인트

```tsx
// 개발자 코드
<FormField>
  <Label>이름</Label>
  <Input />
  <ErrorMessage>필수 입력 항목입니다</ErrorMessage>
</FormField>

// HANUI가 자동 생성
<div>
  <label id="field-1-label" for="field-1-input">이름</label>
  <input
    id="field-1-input"
    aria-labelledby="field-1-label"
    aria-describedby="field-1-error"
    aria-invalid="true"
  />
  <span id="field-1-error" role="alert">
    필수 입력 항목입니다
  </span>
</div>
```

#### 자동 연결 요소

- `label` ↔ `input`의 id/for 자동 매칭
- `aria-describedby` 자동 링크
- 오류 메시지 읽기 지원
- 필드 유효성 상태 자동 전달

#### 중요성

> **접근성 문제의 50% 이상이 Form에서 발생합니다.**
>
> 출처: [Deque Accessibility Audit Trends](https://www.deque.com/)

---

### 4) 인터랙션 상태 자동 처리

**hover, focus, pressed, disabled 등 상태마다 접근성 정보 자동 부여**

#### 자동 처리 ARIA 상태

| 상태                | ARIA 속성       | 적용 시점          |
| ------------------- | --------------- | ------------------ |
| **눌림**            | `aria-pressed`  | 토글 버튼          |
| **펼침/접힘**       | `aria-expanded` | 아코디언, 드롭다운 |
| **숨김**            | `aria-hidden`   | 조건부 렌더링      |
| **실시간 업데이트** | `aria-live`     | 알림, 상태 변경    |
| **로딩**            | `aria-busy`     | 비동기 작업 중     |

#### 효과

- UI 라이브러리의 **일관성** 강화
- 컴포넌트 동작의 **예측 가능성** 향상
- 스크린 리더 사용자 경험 대폭 개선

---

### 🎯 구현 우선순위 결론

HANUI가 시작한다면 다음 순서로 구현:

1. **포커스 관리** (Focus Trap + Restoration + 키보드 네비게이션)
2. **ARIA 자동화** (role + 상태 속성)
3. **Form 연계 자동화** (label 연결 + 에러 메시지)
4. **인터랙션 상태 처리** (aria-pressed, expanded, live 등)

> ✨ 이 조합으로 **Radix UI와 비슷한 임팩트를 바로 낼 수 있습니다.**

## Q2. KRDS와 WCAG 동시 만족 구조

### 결론

**필요하며, 기술적으로 "충돌 없이 통합 가능한 공통 구조"를 만들 수 있습니다.**

---

### 왜 동시에 맞춰야 하나?

| 이유               | 설명                                          |
| ------------------ | --------------------------------------------- |
| **한국 정부 입찰** | KRDS는 의무 기준                              |
| **글로벌 확장**    | WCAG은 국제 표준                              |
| **차별성**         | 둘을 동시에 지원하는 UI 자체가 HANUI의 차별성 |

---

### 두 기준의 차이

| 구분          | WCAG            | KRDS                  |
| ------------- | --------------- | --------------------- |
| **적용 범위** | 국제 표준       | 한국 공공기관         |
| **강조점**    | 장애 범주 전체  | 콘텐츠 구조·국문 기준 |
| **검사 방식** | 정량 + 정성     | 자동화 검사 비중 높음 |
| **업데이트**  | WCAG 2.2 (2023) | KWCAG 2.2 (한국형)    |

**출처**:

- KRDS (한국형 웹 콘텐츠 접근성 지침): https://www.webwatch.or.kr/
- WCAG 2.2: https://www.w3.org/WAI/WCAG22/

---

### 통합 구조: "중앙 접근성 엔진" 설계

HANUI가 제공해야 할 구조:

```
┌─────────────────────────────────────┐
│    HANUI Accessibility Engine       │
├─────────────────────────────────────┤
│                                     │
│  ┌──────────────┐  ┌─────────────┐ │
│  │ WCAG 2.2     │  │ KRDS 2.2    │ │
│  │ 기반 구현    │  │ 기반 검사   │ │
│  └──────────────┘  └─────────────┘ │
│           │               │         │
│           └───────┬───────┘         │
│                   ▼                 │
│         ┌──────────────────┐        │
│         │ 컴포넌트 스키마  │        │
│         │  - Button        │        │
│         │  - Modal         │        │
│         │  - Tabs          │        │
│         │  - Form          │        │
│         └──────────────────┘        │
└─────────────────────────────────────┘
```

---

### 구현 전략 4단계

#### 1) 기본 role/ARIA → WCAG 기준으로 구현

- 모든 컴포넌트에 WCAG 2.2 기준 적용
- 국제 표준을 기본으로 삼아 글로벌 호환성 확보

#### 2) 문자열/label/대체 텍스트 → KRDS 요구사항 자동 검사

- 한국어 콘텐츠 구조 검증
- 필수 대체 텍스트 누락 검사
- 레이블 명확성 체크

#### 3) 콘트라스트 기준 → WCAG AA 이상 자동 체크

- 색상 대비율 자동 계산
- 경고 메시지 개발 도구 제공
- 디자인 토큰에 접근성 기준 내장

#### 4) 컴포넌트 단위 접근성 스키마 제공

각 컴포넌트마다 접근성 체크리스트 자동 생성:

```json
{
  "component": "Button",
  "wcag": {
    "2.1.1": "pass", // Keyboard
    "2.4.7": "pass", // Focus Visible
    "4.1.2": "pass" // Name, Role, Value
  },
  "krds": {
    "perceivable": "pass",
    "operable": "pass",
    "understandable": "pass",
    "robust": "pass"
  }
}
```

---

### 🎯 통합 전략 요약

> **"WCAG 기반 구현 + KRDS 기반 검사"를 통합하는 구조가 가장 적합한 전략입니다.**

**장점**:

- 글로벌 표준 준수
- 한국 공공기관 입찰 대응
- 자동화된 접근성 검사
- 개발자 부담 최소화

## Q3. A11y 브랜드 메시지 전략

### 핵심 포지셔닝

> **A11y는 단순히 장애인 접근성이 아니다.**
>
> **"UX 자동화 + 국제 표준 기반 + 디자인 자유도 보장"**

---

### HANUI 브랜드 메시지 3가지 축

#### 1) A11y = '품질을 자동으로 보장하는 엔진'

**메시지 방향**:

- "접근성이 자동으로 적용되는 UI"
- "설계하지 않아도 표준을 지키는 컴포넌트"
- "누구나 접근성을 잘 적용한 제품을 만들 수 있게 해주는 기술"

**타겟**: 개발팀 리드, CTO, 기술 의사결정자

**참조**: Radix UI가 접근성 중심으로 성공한 모델을 그대로 적용

---

#### 2) A11y = '개발자와 디자이너의 시간을 절약하는 기술'

**브랜드 메시지 예시**:

> "기능에 집중하고, 접근성은 시스템이 책임진다."

> "KRDS·WCAG을 동시에 충족하는 최초의 한국형 UI 엔진."

> "접근성 검사? 자동입니다. 당신은 제품만 만드세요."

**타겟**: 실무 개발자, 프론트엔드 팀

**효과**:

- ⏱️ 개발 시간 단축
- 📋 접근성 체크리스트 자동 완성
- 🎯 비즈니스 로직에 집중 가능

---

#### 3) A11y = '디자인 자유도를 방해하지 않는 기술'

**디자이너가 무조건 좋아하는 메시지**

**브랜드 메시지 예시**:

> "기능은 견고하게, 디자인은 완전히 자유롭게."

> "표준은 기본, 스타일은 100% 커스터마이즈."

> "접근성 때문에 디자인을 포기하지 마세요."

**타겟**: 디자인 팀, 브랜드 디자이너

**차별화**:

- 🎨 완전한 디자인 자유도
- 🔧 Base Layer + Theme Layer 분리
- 🌈 무제한 커스터마이징

---

### 📊 브랜드 메시지 매트릭스

| 타겟            | 핵심 메시지         | 효과               |
| --------------- | ------------------- | ------------------ |
| **개발팀 리드** | 품질 자동 보장 엔진 | 기술 의사결정 설득 |
| **실무 개발자** | 시간 절약 기술      | 도입 의지 강화     |
| **디자이너**    | 디자인 자유도 보장  | 저항감 제거        |
| **경영진**      | KRDS/WCAG 동시 대응 | ROI 명확화         |

---

### 🎯 최종 포지셔닝

```
HANUI = 접근성 자동화 + 디자인 자유도 + KRDS/WCAG 동시 대응
```

**3개 축으로 메시지를 만들면 시장에서 유일한 포지션 확보가 가능합니다.**

## 실행 로드맵

### Phase 1: 핵심 접근성 엔진 구축 (0-3개월)

- [ ] 포커스 관리 시스템 개발
  - Focus Trap
  - Focus Restoration
  - 키보드 네비게이션
- [ ] ARIA 자동화 시스템
  - role 자동 삽입
  - 상태 속성 자동 관리
- [ ] Form 접근성 자동화
  - label 연결
  - 에러 메시지 연동
  - 유효성 검사 ARIA

### Phase 2: KRDS/WCAG 통합 검증 (3-6개월)

- [ ] 중앙 접근성 엔진 설계
- [ ] WCAG 2.2 준수 검증
- [ ] KRDS 2.2 준수 검증
- [ ] 자동 검사 도구 개발
- [ ] 컴포넌트 스키마 문서화

### Phase 3: 브랜드 구축 및 확산 (6-12개월)

- [ ] 접근성 자동화 데모 사이트
- [ ] 개발자 문서 작성
- [ ] 케이스 스터디 (공공기관 + 스타트업)
- [ ] 커뮤니티 빌딩
- [ ] 접근성 인증 획득

### Phase 4: 글로벌 확장 (12개월+)

- [ ] 영문 문서화
- [ ] 해외 접근성 표준 지원
  - ADA (미국)
  - EN 301 549 (EU)
- [ ] 글로벌 컨퍼런스 발표
- [ ] 오픈소스 커뮤니티 성장
