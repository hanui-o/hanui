## 1. 문제 정의

### 디자인 자유도의 딜레마

버튼이든 입력창이든, 최소한의 "형태"는 필요합니다. 그러나:

- **완전한 자유도(unstyled)**: 개발자가 모든 것을 직접 스타일링해야 함
- **완성형 디자인**: 실무에 바로 적용 가능하지만 커스터마이징이 어려움

### Radix UI의 성공 요인

Radix가 전 세계에서 사랑받는 이유는 바로 **이 균형**을 잘 잡았기 때문입니다.

### HANUI의 기회

HANUI는 이 구조를 그대로 가져가면서, **"한국형 디자인 시스템의 장점"**을 더하면 차별화가 생깁니다.

## 2. Base Layer + Theme Layer 구조

### 핵심 아키텍처

버튼의 **기본 형태(base)**는 제공하되, **디자인(theme)**은 원하는 스타일 시스템에 맞춰 갈아끼우는 방식입니다.

---

### ① Base Layer: 디자인 없는 기능 제공

**제공 요소**:

- 버튼의 역할 (`role`, `aria-label`)
- 포커스 트랩
- 키보드 조작
- 이벤트 처리
- 비활성/로딩 상태
- 상태 관리(logic)

**특징**:

- CSS를 거의 넣지 않거나 아주 최소화
- 형태는 없지만 **"작동은 100% 가능"**한 컴포넌트

**예시**:

```tsx
<Button.Root>// 스크린리더 + 포커스링만 있음 // 모양은 없음</Button.Root>
```

---

### ② Theme Layer: 디자인 제공

**디자인 자유도 제공 방식**:

| 방식                                    | 설명                         |
| --------------------------------------- | ---------------------------- |
| **Tailwind preset**                     | Tailwind 기반 빠른 스타일링  |
| **CSS variables 기반 테마**             | 변수만 바꿔서 전체 테마 변경 |
| **Styled-components / Emotion version** | CSS-in-JS 선호 개발자 지원   |
| **기본 KRDS 스타일 세트**               | 공공 프로젝트용 즉시 사용    |

**선택형 제공 방식**:

기본 버튼은 **뼈대뿐**이고, "KRDS 스타일", "모던 스타일", "다크 스타일"을 테마로 선택하는 방식입니다.

**경쟁 우위**:

- Chakra UI 모델보다 더 유연
- MUI보다 훨씬 가볍고 빠름
- Radix보다 더 **"실무형 디자인 제공"**

## 3. 버튼의 기본 모양: 3가지 옵션

### 정답: 유저가 선택하게 해야 함

---

### 옵션 A — 완전 Unstyled (Radix 스타일)

```tsx
<Button.Base>
```

**제공 요소**:

- 사각형조차 없음
- 포커스링만 있음
- 개발자가 직접 스타일링

**적합한 사용자**:

- 완전한 커스터마이징을 원하는 개발자
- 자체 디자인 시스템을 구축하는 팀

---

### 옵션 B — 최소 모양 제공

```tsx
<Button.Minimal>
```

**제공 요소**:

- 사각형 형태
- 기본 여백 + 정렬
- 색상 없음
- 가이드라인만 지켜진 형태

**적합한 사용자**:

- 빠른 프로토타이핑이 필요한 개발자
- 최소한의 구조만 필요한 프로젝트

---

### 옵션 C — 테마 스타일 적용

```tsx
<Button theme="krds" />
<Button theme="modern" />
<Button theme="dark" />
```

**제공 요소**:

- 완성된 디자인
- 공공사업용 KRDS 스타일
- 모던 SaaS 스타일
- 다크모드 지원

**적합한 사용자**:

- 공공사업 입찰 프로젝트
- 스타트업의 빠른 MVP 개발
- 기업 내부 관리 시스템

> **공공사업 + 스타트업 모두에게 적합한 구조**

## 4. HANUI가 Radix보다 강해질 포인트

### Radix의 한계

Radix는 UI 디자인을 전혀 제공하지 않는 것이 단점입니다.

실무에서는 **Radix + Tailwind + shadcn** 조합으로 복잡해집니다.

---

### HANUI의 차별화 공식

```
HANUI = Radix의 접근성 + shadcn의 스타일 + KRDS의 표준화
```

**세부 분석**:

| 요소                | 제공 가치                      |
| ------------------- | ------------------------------ |
| **Radix의 접근성**  | 세계 최고 수준의 A11y 자동화   |
| **shadcn의 스타일** | 즉시 사용 가능한 실무형 디자인 |
| **KRDS의 표준화**   | 한국 공공서비스 표준 준수      |

> ✨ **이 조합이 바로 HANUI가 전 세계 시장에서 대체 불가능해지는 지점입니다.**

---

### 실무 적용 시나리오

#### 시나리오 1: 공공기관 프로젝트

```tsx
import { Button } from '@hanui/react';

// KRDS 테마 자동 적용
<Button theme="krds">신청하기</Button>;
// 접근성 자동 완성
// 공공서비스 디자인 가이드 준수
// 입찰 요구사항 충족
```

#### 시나리오 2: 스타트업 MVP

```tsx
import { Button } from '@hanui/react';

// 모던 SaaS 스타일
<Button theme="modern" variant="primary">
  Get Started
</Button>;
// 빠른 개발
// 세련된 디자인
// 브랜드 커스터마이징 가능
```

#### 시나리오 3: 완전 커스텀

```tsx
import { Button } from '@hanui/react';

// Base 컴포넌트 + 자체 스타일
<Button.Base className="my-custom-styles">
// 완전한 디자인 자유도
// 접근성은 자동 보장
// 키보드 네비게이션 자동 처리
```

## 5. 핵심 전략 요약

### 🎯 디자인 자유도를 높이면서도 실무에서는 '빠르게 쓸 수 있는' 두 레이어 구조

#### 1. Base Layer

- 접근성 + 로직 100% 제공
- 디자인은 최소화 또는 제로
- 모든 기능은 완벽하게 작동

#### 2. Theme Layer

- 디자인 여러 버전 제공
- 선택형 테마 시스템
- CSS Variables / Tailwind / Styled-components 모두 지원

#### 3. KRDS 스타일 세트

- 공공서비스 시장 특화
- 한국형 디자인 시스템 표준
- 입찰 요구사항 자동 충족

#### 4. Custom Theme SDK

- 기업마다 브랜드 맞춤 디자인 가능
- Design Token 시스템 연동
- Figma → Code 자동화 지원

---

### 핵심 가치 제안

이렇게 하면 **"자유도 + 실무성 + 공공성"**을 모두 잡습니다.

| 경쟁사            | HANUI의 차별화                          |
| ----------------- | --------------------------------------- |
| **vs Radix**      | 실무형 디자인 + KRDS 표준 제공          |
| **vs MUI**        | 더 가벼움 + 더 높은 커스터마이징 자유도 |
| **vs Ant Design** | 글로벌 접근성 + 한국 공공표준           |
| **vs shadcn**     | 공공서비스 특화 + 엔터프라이즈 지원     |
